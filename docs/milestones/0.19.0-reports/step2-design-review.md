## 审查维度

### 1. 产品定位一致性 ✅

- 单用户、自托管：观测与诊断功能是对系统自身的监控，不涉及多用户或团队协作，符合单用户定位。
- 不引入多用户/RBAC 概念：功能仅涉及系统指标和健康检查，无权限概念。
- 数据不经过浏览器中转：指标通过 API 查询，前端仅展示，不涉及文件传输。

### 2. 架构一致性 ✅

- 单二进制架构不变，观测逻辑在 rex-hub 内实现（新增 metrics.rs、health.rs）。
- 复用现有 SQLite 数据库，新增 metrics 表，不引入新存储。
- 前端按功能域组织：新增 api/health.ts、api/metrics.ts，以及仪表盘和工作区的性能组件。

### 3. 安全性 ✅

- 指标数据不包含敏感信息（仅延迟、吞吐量、错误率等聚合数据）。
- 健康 API 仅返回系统级指标（CPU、内存、磁盘、数据库大小），不暴露用户数据或令牌。
- API 需要认证（注册在 protected_routes 中）。

### 4. 子任务拆分粒度 ✅

4. 仪表盘性能概览前端
5. 实时性能指标展示
6. 测试验证

粒度合理，每个子任务 1-2 个 commit。

### 5. 产品文档边界 ✅

- 观测与诊断未在 PRODUCT.md 中明确定义，属于运维辅助功能。
- 不做第三方监控集成（合理，避免依赖外部服务）。

## 发现问题

### 🟡 指标采集可能阻塞主业务

**位置**：子任务1，指标数据模型与采集后端

**问题**：里程碑文档中未明确指标采集应为异步非阻塞操作。如果采集同步进行，可能影响主业务响应时间。

**建议**：在后端实现中，使用异步方式写入指标（例如通过 tokio::spawn 或专门的后台任务），避免阻塞请求处理链路。

### 🟡 健康 API 可能暴露过多系统细节

**位置**：子任务2，系统健康 API

**问题**：健康 API 包含磁盘使用率等系统指标，在某些容器化环境中可能不准确或具有误导性。

**建议**：在 API 文档中明确说明这些指标仅供参考，并考虑仅在非容器化部署时启用某些指标（如磁盘使用率）。或者提供一个开关以关闭敏感系统指标。

### 🟡 性能图表依赖 SVG 实现可能复杂度较高

**位置**：子任务4，仪表盘性能概览前端

**问题**：里程碑文档中提到使用 SVG 实现性能图表，这可能增加前端开发复杂度。

**建议**：考虑使用更简单的文本或进度条展示关键指标（如平均延迟、错误率），或如果必须使用图表，则使用轻量级的图表库（如 Chart.js）但需评估其引入的 bundle 大小。然而，产品文档中未禁止使用第三方库，但强调“不做第三方监控集成”指的是后端监控系统，前端使用小型图表库可能是可接受的。

### 🟡 指标查询索引需要验证

**位置**：子任务1，指标数据模型与采集后端

**问题**：里程碑文档中提到索引 `idx_metrics_resource_type` 和 `idx_metrics_recorded_at`，但需要确认这些索引是否足以支持聚合查询和时间线查询的性能。

**建议**：在实施后进行查询性能测试，必要时添加复合索引（如 `(resource_id, metric_type, recorded_at)`）。

### 🟢 可选改进：添加指标采集速率限制

**位置**：子任务1

**建议**：为了防止在高频操作下产生过多指标数据，可以在采集端添加简单的速率限制（如每秒最多采集 N 个相同类型的指标）。
