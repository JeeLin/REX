# 0.17.0: 全局查询功能

## Context

0.16.0 完成了 SQL AI 助手（后端 AI 配置存储 + 代理 API SSE 流式、前端 AI 助手面板、快捷操作 + 代码复制集成）。PRODUCT.md 描述了 SQL 控制台的全局查询功能（Ctrl+Shift+Q），允许用户同时选中多个数据库标签执行跨库查询，仅支持相同方言的数据库。目前代码库中尚未实现此功能。

版本类型：minor（新功能）

## 产品边界

**做什么：**
- 后端：接受多个数据库连接信息，并行执行相同 SQL 查询
- 前端：模态弹窗界面，支持多选数据库标签，显示执行进度和结果
- 前端：仅允许选择相同方言的数据库进行跨库查询
- 前端：展示合并后的结果，支持分页、导出等常规查询功能
- 前端：提供取消按钮，支持中途停止查询执行

**不做什么：**
- 不支持不同方言数据库之间的真正联合查询（如 MySQL 和 PostgreSQL 的 JOIN）
- 不自动转换 SQL 方言（用户需确保所选数据库使用相同方言）
- 不保存跨库查询为查询文件（暂仅支持一次性执行）
- 不在查询过程中自动处理事务（每个数据库独立执行）

## 子任务详细设计

### 1 全局查询后端 API 实现

**功能目标：**
实现 POST /api/sql/global-query 端点，接收多个数据库连接参数和 SQL 语句，在后台并行执行查询，返回合并结果。

**文件结构：**
```text
crates/rex-hub/src/
├── sql.rs              修改：添加全局查询处理函数
├── routes.rs           修改：注册 /api/sql/global-query 路由
crates/rex-common/src/
├── db.rs               修改：添加多连接查询辅助函数
```

**数据模型：**
无需新表，复用现有连接信息。

**API 设计：**
```text
POST /api/sql/global-query
Body: {
  "databases": [
    { "id": "conn1", "name": "db1", "type": "mysql", "host": "...", "port": 3306, "username": "...", "password": "encrypted", "database": "mydb" },
    { "id": "conn2", "name": "db2", "type": "mysql", "host": "...", "port": 3306, "username": "...", "password": "encrypted", "database": "mydb" }
  ],
  "sql": "SELECT * FROM users WHERE age > 18",
  "limit": 100,
  "offset": 0
}
Response: 流式响应（application/json-stream）
  data: {"type":"start","totalConnections":2}
  data: {"type":"result","connectionId":"conn1","data":[{"id":1,"name":"John"}],"rowCount":1}
  data: {"type":"result","connectionId":"conn2","data":[{"id":2,"name":"Jane"}],"rowCount":1}
  data: {"type":"progress","completed":2,"total":2}
  data: {"type":"done","totalRows":2}
  data: {"type":"error","connectionId":"conn1","message":"Connection timeout"}
```

**后端流程：**
1. 验证所有连接 ID 是否有效且属于同一方言
2. 为每个连接创建独立的数据库连接
3. 并行执行 SQL 查询（使用 tokio::spawn 或类似机制）
4. 收集每个连接的结果，包含错误信息
5. 按顺序流式返回结果（先开始 → 各连接结果 → 完成/错误）
6. 确保连接池正确释放

**测试标准：**
- 单个数据库查询正常工作
- 多个相同方言数据库并行查询
- 不同方言数据库返回错误
- 部分连接失败时继续处理其他连接
- 取消请求时及时停止所有查询

**提交信息：**
```
feat: add global query API endpoint
```

### 2 全局查询前端模态弹窗 UI

**功能目标：**
实现 Ctrl+Shift+Q 快捷键触发的模态弹窗，包含数据库多选列表、SQL 编辑器、执行按钮、进度条和结果展示区。

**文件结构：**
```text
packages/rex-console-web/src/pages/
├── Sql.vue                 修改：添加全局查询按钮和事件处理
src/
├── components/
│   └── GlobalQueryModal.vue 新增：全局查询模态组件
├── composables/
│   └── useGlobalQuery.ts   新增：全局查询逻辑封装
```

**组件设计：**
```vue
<!-- GlobalQueryModal.vue -->
<template>
  <div class="modal-overlay" @click.self="close">
    <div class="modal-content">
      <div class="modal-header">
        <span>⊞ 全局查询</span>
        <button @click="close">×</button>
      </div>
      
      <div class="modal-body">
        <!-- 数据库选择 -->
        <div class="db-selector">
          <label>数据库：</label>
          <div class="db-checkbox-group">
            <label v-for="db in databases" :key="db.id" class="db-checkbox">
              <input type="checkbox" :value="db.id" v-model="selectedDatabases" :disabled="!isCompatible(db.type)" />
              <span>{{ db.name }} ({{ db.type }})</span>
            </label>
          </div>
          <small class="hint">仅支持相同方言的数据库</small>
        </div>
        
        <!-- SQL 编辑器 -->
        <div class="sql-editor">
          <textarea v-model="sqlQuery" placeholder="输入 SQL 查询语句..." />
          <div class="editor-toolbar">
            <button @click="execute">执行 (Ctrl+Enter)</button>
            <button @click="cancel" :disabled="!isExecuting">取消</button>
            <progress :value="progress" :max="100" v-if="isExecuting"></progress>
          </div>
        </div>
        
        <!-- 结果展示 -->
        <div class="results-tabs" v-if="results.length > 0">
          <div class="tab-bar">
            <button v-for="(result, index) in results" :key="index" 
                    :class="{ active: activeTab === index }"
                    @click="activeTab = index">
              {{ result.connectionName }} ({{ result.rowCount }} 行)
            </button>
          </div>
          <div class="tab-content">
            <result-view v-if="results[activeTab]" :data="results[activeTab].data" :columns="results[activeTab].columns" />
          </div>
        </div>
        <div v-else class="empty-state">暂无结果</div>
      </div>
    </div>
  </div>
</template>
```

**交互设计：**
- 按下 Ctrl+Shift+Q 打开模态框
- 模态框打开时自动聚焦到 SQL 编辑器
- 勾选数据库时自动检查方言兼容性，不兼容的置灰且不可选
- SQL 编辑器支持基本的语法高亮（可复用现有组件）
- 按下 Ctrl+Enter 执行查询
- 执行过程中显示总体进度条和取消按钮
- 结果以选项卡形式展示，每个选项卡对应一个数据库连接
- 每个结果保持原有的查询结果视图功能（排序、导出等）

**测试标准：**
- 快捷键 Ctrl+Shift+Q 打开/关闭模态框
- 数据库列表正确显示当前工作空间的连接
- 不同方言的数据库无法同时选中
- 执行按钮在未选择数据库或 SQL 为空时禁用
- 执行过程中显示实时进度
- 结果正确显示并在选项卡之间切换
- 取消按钮在执行中可用，点击后停止所有查询

### 3 数据库方言验证和多选逻辑

**功能目标：**
实现数据库方言检查逻辑，确保只能选择相同方言的数据库进行跨库查询。

**文件结构：**
```text
src/
├── composables/
│   └── useGlobalQuery.ts   修改：添加方言验证逻辑
```

**实现细节：**
1. 定义方言映射：mysql → mysql, postgresql → postgresql, 等等
2. 当用户选择第一个数据库时，记录其方言
3. 后续选择的数据库必须具有相同的方言，否则自动取消选择并提示
4. 提供“全选 compatible”按钮，仅选中方言相同的数据库
5. 在提交前再次验证所有选中数据库方言一致

**测试标准：**
- 选择 MySQL 数据库后，只能选择其他 MySQL 实例
- 选择 PostgreSQL 后，只能选择其他 PostgreSQL 实例
- 混合选择时自动纠正为单一类型
- 方言不匹配时给出明确提示

**提交信息：**
```
feat: implement database dialect validation for multi-select
```

### 4 并行查询执行和结果合并

**功能目标：**
在后端实现真正的并行查询执行，并在前端正确合并和显示结果。

**文件结构：**
```text
crates/rex-hub/src/
├── sql.rs                修改：实现并行查询逻辑
crates/rex-common/src/
├── db.rs                 修改：添加并行查询辅助函数
```

**后端实现：**
- 使用 tokio::join! 或 futures::future::join_all 并行执行查询
- 每个查询在独立的任务中运行，互不阻塞
- 设置超时机制防止单个查询卡死
- 收集每个查询的结果（成功或错误）
- 按固定顺序（按照请求中的数据库顺序）发送结果流

**前端处理：**
- 将流式响应解析为消息流
- 按 connectionId 分类存储结果
- 每收到一个结果就更新对应的选项卡数据
- 保持结果的原始格式（列名、数据类型等）

**测试标准：**
- 两个相同查询在不同数据库上执行时间应显著小于串行执行时间
- 大结果集不会阻塞 UI 线程（使用虚拟滚动或分页）
- 查询超时时正确返回错误而不影响其他查询
- 结果顺序与请求中的数据库顺序一致

**提交信息：**
```
feat: implement parallel query execution and result handling
```

### 6 错误处理和部分失败场景

**功能目标：**
优雅地处理各种错误情况，包括部分数据库连接失败、查询超时、语法错误等。

**文件结构：**
```text
src/
├── composables/
│   └── useGlobalQuery.ts   修改：增强错误处理逻辑
```

**错误类型：**
1. 连接失败：网络不可达、认证失败等
2. 查询执行错误：SQL 语法错误、表不存在等
3. 超时错误：查询执行超过设定时间
4. 部分结果：一些数据库成功，一些失败

**前端处理：**
- 在结果选项卡中显示错误状态而非数据
- 错误信息友好提示（如“连接超时”、“访问被拒绝”等）
- 成功的结果仍然正常显示
- 总体状态显示部分成功/失败
- 提供重试单个失败查询的选项

**后端处理：**
- 每个查询任务独立捕获异常
- 将错误信息封装在结果流中返回
- 不让单个查询的失败影响其他查询的执行
- 记录详细错误日志便于排查

**测试标准：**
- 单个数据库连接失败时，其他数据库仍能正常查询
- SQL 语法错误只返回相应数据库的错误，不影响其他数据库
- 超时配置生效，超时后返回错误而非永久等待
- 错误信息准确指向具体是哪个数据库出问题
- 部分成功时，用户能清楚看到哪些成功哪些失败

**提交信息：**
```
feat: enhance error handling for partial failures
```
