# 0.15.0: 跨连接文件传输

## Context

0.14.0 完成了核心面板精化（SQL 导出、右键菜单、仪表盘徽章、全局连接菜单）。现有的传输基础设施已就绪：`TransferManager` 任务队列、`FileConnector` trait、`LocalConnector`/`SftpConnector` 实现、API 路由（`/api/transfers`）、前端 UI 组件（`TransferItem.vue`、`useTransferQueue.ts`）。但 `TransferManager` 只管理任务状态，**不执行实际传输**——调用 `create_transfer` 后任务停留在 `pending` 状态。

本里程碑实现传输执行引擎，打通跨连接文件传输的完整链路。

版本类型：minor（功能增强）

## 产品边界

**做什么：**
- 传输执行引擎：读取源 connector → 写入目标 connector → 更新进度 → 完成/失败
- Hub 连接器池：根据 resource_id 查找已活跃的 WebSocket 连接，创建对应的 FileConnector
- 端点解析：从数据库读取资源凭据，构建可用的 FileConnector
- 单文件跨连接传输（SSH→SSH、SSH→本地、本地→SSH）

**不做什么：**
- 不实现递归目录传输（下个里程碑）
- 不新增协议的 FileConnector（SSH/SFTP + Local 已覆盖核心场景）
- 不修改前端 UI（TransferItem.vue、useTransferQueue.ts 已就绪）
- 不引入传输调度器或并发控制
- 不实现冲突处理（目标已存在时直接覆盖，后续里程碑增加冲突检测 UI）
- 不实现 `sftp` + 连接参数回退路径（仅支持 `sftp` + `resource_id` 模式，即复用已活跃的 WebSocket 连接）
- 本里程碑先用全量读写（`source.read()` → `target.write()`），流式分块传输作为后续优化

## 子任务详细设计

### 1 传输执行引擎

**功能目标：**
在 `rex-transfer` crate 中新增 `TransferExecutor`，负责实际执行文件传输：从源 `FileConnector` 读取数据，写入目标 `FileConnector`，更新 `TransferManager` 中的进度。

**文件结构：**
```text
crates/rex-transfer/src/
├── lib.rs              不修改
├── task.rs             修改：TransferManager 新增 spawn_executor 方法
├── executor.rs         新增：传输执行逻辑
├── local.rs            不修改
└── sftp.rs             不修改
```

**接口设计：**
```rust
// executor.rs
use std::sync::Arc;
use crate::{FileConnector, task::{TransferManager, TransferStatus}};

/// 执行单个传输任务
/// - 从 source connector 读取文件
/// - 写入 target connector
/// - 通过 manager 更新进度
/// - 完成/失败时更新状态
pub async fn execute_transfer(
    manager: Arc<TransferManager>,
    task_id: String,
    source: Arc<dyn FileConnector>,
    target: Arc<dyn FileConnector>,
    source_path: std::path::PathBuf,
    target_path: std::path::PathBuf,
) {
    // 1. 设置状态为 Running
    // 2. source.metadata() 获取文件大小
    // 3. source.read() 读取全部内容
    // 4. target.write() 写入目标
    // 5. 更新进度（total = size, transferred = size）
    // 6. 设置状态为 Completed
    // 任何步骤失败 → 设置状态为 Failed(error_message)
}
```

**交互设计：**
无前端交互变更。

**后端流程：**
1. 从 `TransferManager` 获取任务
2. 将状态从 `Pending` 设为 `Running`
3. 调用 `source.read(source_path)` 读取文件字节
4. 调用 `target.write(target_path, bytes)` 写入目标
5. 每次写入后更新 `progress`
6. 成功 → `Completed`，失败 → `Failed(reason)`

**测试标准：**
- `execute_transfer` 单元测试：两个 `LocalConnector` 之间的文件传输
- 验证进度更新：`Pending → Running → Completed`
- 验证失败处理：源文件不存在时状态为 `Failed`

**提交信息：**
```

```

### 2 Hub 连接器池 + 端点解析

**功能目标：**
在 `rex-hub` 中实现连接器池——根据 `TransferEndpoint` 中的 `resource_id` 或连接参数，创建对应的 `FileConnector` 实例。

**文件结构：**
```text
crates/rex-hub/src/
├── transfer.rs         修改：新增 resolve_connector 函数
├── ws.rs               修改：Connections 新增获取活跃连接的方法
├── files.rs            不修改
```

**接口设计：**
```rust
// transfer.rs 新增
use std::sync::Arc;
use rex_transfer::FileConnector;

/// 根据端点解析为 FileConnector
/// - connector_type="local" → LocalConnector（基于 Hub 本地路径）
/// - connector_type="sftp" → SftpConnector（通过 SSH 连接）
async fn resolve_connector(
    endpoint: &TransferEndpoint,
    connections: &Connections,
    db: &Database,
) -> Result<Arc<dyn FileConnector>, String>
```

**端点解析规则：**

| connector_type | 解析方式 |
|---|---|
| `local` | `LocalConnector::new(endpoint.path)` — Hub 本地文件系统 |
| `sftp` + resource_id | 从活跃 WebSocket 连接中获取（`Connections` 中查找） |

**后端流程：**
1. `create_transfer` API 被调用
2. 验证端点参数（已有）
3. 调用 `resolve_connector` 解析 source 和 target
4. 调用 `TransferExecutor::execute_transfer` 启动 tokio::spawn
5. 返回 201 + 任务信息

**测试标准：**
- 解析 `connector_type="local"` 返回 `LocalConnector`
- 解析 `connector_type="sftp"` 但无活跃连接时返回错误
- 参数验证：未知 connector_type 返回错误

**提交信息：**
```
feat: add connector resolver for transfer endpoints
```

### 3 集成传输启动到 API 路由

**功能目标：**
将传输执行集成到 `create_transfer` API 处理流程中——创建任务后立即启动异步执行。

**文件结构：**
```text
crates/rex-hub/src/
├── transfer.rs         修改：create_transfer 中启动 executor
├── routes.rs           不修改
```

**接口设计：**
无 API 变更，现有接口保持不变。

**后端流程：**
1. `create_transfer` handler：
   - 验证端点参数（已有）
   - 解析 source/target connector（新增）
   - 创建传输任务（已有）
   - `tokio::spawn` 执行传输（新增）
   - 返回 201 + 任务信息（已有）
2. 前端通过轮询 `GET /api/transfers/:id` 或 `GET /api/transfers` 获取进度更新

**测试标准：**
- 创建本地→本地传输后，目标路径出现源文件
- 任务状态从 `pending` 变为 `completed`
- 创建传输时源文件不存在，状态变为 `failed`

**提交信息：**
```
feat: integrate transfer executor into create_transfer API
```
