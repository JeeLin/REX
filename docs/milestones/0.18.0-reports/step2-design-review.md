# 步骤2：设计核对报告

## 审查对象

- 里程碑文档：`docs/milestones/0.18.0-配置备份与恢复.md`
- 产品文档：`docs/PRODUCT.md`
- 项目约定：`CLAUDE.md`

## 审查维度

### 1. 产品定位一致性 ✅

- 单用户、自托管：备份/恢复是个人数据管理能力，完全符合
- 不引入多用户/RBAC 概念
- 数据不经过浏览器中转（导出直接下载 JSON，导入上传文件）

### 2. 架构一致性 ✅

- 单二进制架构不变，备份逻辑在 rex-hub 内实现
- 不涉及进程模型变更
- 复用现有 SQLite 数据库，不引入新存储

### 3. 安全性 ✅

- Agent token 原始值不备份（只备份 agent_token_hash）
- 加密使用 PBKDF2-SHA256 + AES-256-GCM
- 导出/导入 API 需要认证
- 设计核对点包含文件大小限制

### 4. 子任务拆分粒度 ✅

5 个子任务，每个聚焦一个完整功能点：
1. 配置导出后端 API
2. 配置导入后端 API
3. 设置页备份面板前端 UI
4. 导入预览与冲突处理
5. 测试验证

粒度合理，每个子任务 1-2 个 commit。

### 5. 产品文档边界 ✅

- 备份/恢复未在 PRODUCT.md 中明确定义，属于运维辅助功能
- 不做自动定时备份（合理，避免过度设计）
- 不做远程存储备份（合理，保持简单）
- 不备份审计日志（合理，审计日志是运行记录）

## 发现问题

### 🟡 合并策略命名混淆

**位置**：子任务2，合并策略定义

`merge` 策略的实际行为是"已存在的跳过，不存在的创建"，与通常理解的 merge（合并/覆盖字段）不同。`skip` 和 `merge` 行为完全相同。

**建议**：
- `merge` → `skip_existing`（更精确）
- 或保留 `merge` 但调整其语义为"合并字段"（需要更复杂实现）

### 🟡 数据模型缺少 nonce 字段

**位置**：子任务1，数据模型定义

`BackupFile` 结构体定义中没有 `nonce` 字段，但加密流程（子任务2）明确提到需要 nonce。需要在子任务1的数据模型中添加。

### 🟡 子任务1内部矛盾

**位置**：子任务1，后端流程第5点

"不加密时直接返回 JSON；加密流程在子任务 2 中设计" 与功能目标中的"支持可选密码加密"矛盾。加密应该在子任务1中完整实现。

## 结论

**✅ 通过**（有 3 个小问题，需在开发前修正）

里程碑文档整体设计合理，与产品定位一致，安全性考虑充分。3 个小问题均为措辞/细节层面，不影响设计方向，可在开发阶段修正。
