# 0.27.2: HTTPS/TLS 修复

## Context

0.27.1 完成了 UI 设计优化与无障碍（Toast/ConfirmDialog 统一、i18n、ARIA、CSS 变量）。本里程碑修复 HTTPS/TLS 层面的关键问题——Agent 无法连接使用自签名证书的 Hub、Agent 缺少 TLS 证书信任配置、ACME 错误处理不健壮。

当前 TLS 实现存在以下问题：
- Agent 使用 `tokio-tungstenite::connect_async` 连接 `wss://` 时无法接受 Hub 的自签名证书
- Agent 使用 `reqwest::Client::new()` 下载更新时同样不信任自签名证书
- Agent 没有 `--ca-cert` 或 `--insecure` 等 TLS 信任配置选项
- HTTP-01 challenge 服务器端口硬编码为 80，Docker 环境易冲突
- 自签名证书过期后无自动续期/重新生成机制
- `parse_cert_not_after` DER 解析逻辑脆弱

版本类型：patch（bug 修复），版本号 0.27.2。

## 产品边界

**做什么：**
- Agent 支持信任 Hub 的自签名/自定义 CA 证书
- Agent 提供 `--ca-cert`、`--insecure` TLS 配置选项
- Agent 自动更新下载使用相同 TLS 信任配置
- HTTP-01 challenge 服务器端口可配置
- 自签名证书过期后自动重新生成
- ACME 驱动失败时提供状态反馈

**不做什么：**
- 不修改 TLS 1.3 协议选择
- 不新增 ACME 证书提供者（仅 Let's Encrypt）
- 不实现证书热加载（仍需重启 Hub）
- 不修改 Agent 与 Hub 的 WebSocket 协议

## 子任务详细设计

### 1 Agent TLS 证书信任配置

- **功能目标**
  Agent 支持通过配置文件、CLI 参数或环境变量指定 TLS 信任设置：自定义 CA 证书（`--ca-cert`）、信任自签名证书（`--insecure`，跳过证书验证）。适用于 Hub 使用自签名证书或内部 CA 签发证书的部署场景。

- **文件结构**
  - 修改：`crates/rex-common/src/cli.rs` — 添加 `--ca-cert`、`--insecure` CLI 参数
  - 修改：`crates/rex-agent/src/config.rs`（如存在）— 添加 `ca_cert`、`insecure` 配置字段
  - 新增：`crates/rex-common/src/tls_client.rs` — 构建 `reqwest::ClientBuilder` 和 `tokio_tungstenite` 连接器的 TLS 配置

- **接口设计**
  CLI 参数：
  ```
  --ca-cert <path>    CA 证书文件路径（PEM 格式，支持多个）
  --insecure          跳过 TLS 证书验证（仅开发/测试环境）
  ```

  环境变量：
  - `REX_CA_CERT=/path/to/ca.pem`
  - `REX_INSECURE=true`

  配置文件（agent.yaml）：
  ```yaml
  tls:
    ca_cert: /path/to/ca.pem
    insecure: false
  ```

  优先级：CLI > 环境变量 > 配置文件

- **交互设计**
  - 使用 `--insecure` 时，Agent 启动日志输出 WARNING：`TLS certificate verification disabled — do not use in production`
  - 使用 `--ca-cert` 时，Agent 启动日志输出：`loading custom CA certificate: /path/to/ca.pem`

- **后端流程**
  1. 解析配置，合并 CLI、环境变量、配置文件
  2. 如果 `--insecure`：构建不验证证书的 `reqwest::Client` 和 `Connector`
  3. 如果 `--ca-cert`：读取 CA 证书，构建信任自定义 CA 的 TLS 配置
  4. 将 TLS 配置传递给 WebSocket 连接和 HTTP 客户端

- **测试标准**
  - 单元测试：配置解析（ca_cert、insecure、优先级）
  - 单元测试：`tls_client` 模块构建不同 TLS 配置的 reqwest Client
  - 单元测试：ca_cert 文件不存在时的错误处理

- **提交信息**
  `feat: add TLS trust configuration options for Agent (--ca-cert, --insecure)`

### 2 Agent 自定义 TLS 配置集成

- **功能目标**
  将子任务1 创建的 TLS 配置应用到 Agent 的 WebSocket 连接（`tokio-tungstenite`）和 HTTP 客户端（`reqwest`），使 Agent 能连接使用自签名或自定义 CA 证书的 Hub。

- **文件结构**
  - 修改：`crates/rex-agent/src/ws.rs` — WebSocket 连接使用自定义 TLS 配置
  - 修改：`crates/rex-agent/src/client.rs` — `HubClient` 接受自定义 `reqwest::Client`
  - 修改：`crates/rex-agent/src/bin/rex-agent.rs`（或 main）— 构建 TLS 客户端并传入

- **接口设计**
  ```rust
  // tls_client.rs
  pub fn build_reqwest_client(ca_cert: Option<&Path>, insecure: bool) -> Result<reqwest::Client>
  pub fn build_ws_connector(ca_cert: Option<&Path>, insecure: bool) -> Result<Connector>
  ```

- **交互设计**
  - Agent 连接使用自签名证书的 Hub 时，日志：`TLS: connecting to hub with custom CA trust` 或 `TLS: certificate verification disabled`
  - 更新下载使用相同的 TLS 配置，不再使用默认 `reqwest::Client::new()`

- **后端流程**
  1. `HubClient::new()` 改为接受 `reqwest::Client` 而非创建默认客户端
  2. `AgentWs::new()` 接受 WebSocket 连接器参数
  3. `connect_async` 使用自定义 `Connector` 替代默认连接器
  4. `perform_update` 中使用共享的 `reqwest::Client`

- **测试标准**
  - 单元测试：`HubClient` 使用自定义 Client 构建
  - 单元测试：WebSocket URL 生成（wss:// 保持不变）
  - 单元测试：insecure 模式下 reqwest Client 配置验证

- **提交信息**
  `fix: integrate custom TLS config into Agent WebSocket and HTTP clients`

### 3 Hub HTTP-01 challenge 端口可配置

- **功能目标**
  HTTP-01 challenge 服务器端口从硬编码 80 改为可配置，默认保持 80。适用于 Docker 环境中端口 80 被占用或无法绑定的场景。

- **文件结构**
  - 修改：`crates/rex-hub/src/config.rs` — `AcmeConfig` 添加 `http_port` 字段
  - 修改：`crates/rex-hub/src/bin/rex-hub.rs` — 使用配置端口绑定 HTTP-01 challenge 服务器
  - 修改：`crates/rex-common/src/cli.rs` — 添加 `--acme-http-port` 参数

- **接口设计**
  配置文件：
  ```yaml
  acme:
    domain: hub.example.com
    email: admin@example.com
    http_port: 8080  # 默认 80
  ```

  环境变量：`REX_ACME_HTTP_PORT=8080`
  CLI：`--acme-http-port 8080`

- **交互设计**
  - 绑定失败时日志输出明确错误：`failed to bind HTTP-01 challenge server on port 80: Address already in use`
  - 绑定成功时日志输出：`HTTP-01 challenge server listening on port 8080`

- **后端流程**
  1. `AcmeConfig` 增加 `http_port: u16` 字段，默认 80
  2. 环境变量 `REX_ACME_HTTP_PORT` 覆盖
  3. CLI `--acme-http-port` 最高优先级
  4. 绑定失败从静默日志改为明确错误提示（但不阻止 Hub 启动）

- **测试标准**
  - 单元测试：`AcmeConfig` 默认 `http_port = 80`
  - 单元测试：环境变量覆盖 `http_port`
  - 单元测试：CLI 参数覆盖

- **提交信息**
  `fix: make HTTP-01 challenge server port configurable`

### 4 自签名证书过期自动重新生成

- **功能目标**
  当缓存的自签名证书已过期或即将过期（≤7 天）时，Hub 自动重新生成并替换。避免证书过期导致 TLS 握手失败。

- **文件结构**
  - 修改：`crates/rex-hub/src/self_signed.rs` — 添加过期检测和自动续期逻辑
  - 修改：`crates/rex-hub/src/bin/rex-hub.rs` — 使用更新后的自签名逻辑

- **接口设计**
  ```rust
  /// 加载或生成自签名证书，过期时自动重新生成
  pub fn load_or_generate_self_signed(
      data_dir: &Path,
      sans: &[String],
  ) -> Result<SelfSignedCert>
  ```

- **交互设计**
  - 证书有效：日志 `loaded existing self-signed certificate (expires in 362 days)`
  - 证书即将过期：日志 `self-signed certificate expiring soon (3 days), regenerating`
  - 证书已过期：日志 `self-signed certificate expired, regenerating`

- **后端流程**
  1. 加载磁盘上的自签名证书
  2. 解析 notAfter 字段，判断是否过期或 ≤7 天过期
  3. 如果过期/即将过期：生成新证书，覆盖旧文件
  4. 如果加载失败（文件缺失/损坏）：生成新证书
  5. 返回有效证书

- **测试标准**
  - 单元测试：有效证书直接加载
  - 单元测试：过期证书触发重新生成
  - 单元测试：文件损坏触发重新生成
  - 单元测试：证书文件缺失触发生成

- **提交信息**
  `fix: auto-regenerate self-signed certificate when expired or expiring`

### 5 ACME 错误处理增强 + TLS 状态 API 改进

- **功能目标**
  ACME 驱动失败时提供更清晰的状态反馈；TLS 状态 API 返回更详细的证书信息（过期时间、ACME 错误原因）。

- **文件结构**
  - 修改：`crates/rex-hub/src/acme.rs` — ACME 驱动失败重试 + 状态追踪
  - 修改：`crates/rex-hub/src/settings.rs` — TLS 状态 API 增强

- **接口设计**
  `GET /api/settings/tls` 响应增强：
  ```json
  {
    "mode": "acme-domain",
    "domain": "hub.example.com",
    "cert_ready": false,
    "cert_expires_at": null,
    "cert_issuer": "Let's Encrypt",
    "port_80_required": true,
    "acme_status": "requesting",
    "acme_error": null
  }
  ```

- **交互设计**
  - ACME 申请中：`acme_status: "requesting"`
  - ACME 证书就绪：`acme_status: "ready"`
  - ACME 失败：`acme_status: "error"`, `acme_error: "rate limit exceeded"`

- **后端流程**
  1. ACME 驱动任务追踪状态（通过 ACME stream 事件）
  2. 失败时记录错误原因，不静默吞错
  3. TLS 状态 API 返回 ACME 状态和错误信息
  4. ACME stream 结束后尝试重建（最多 3 次重试）

- **测试标准**
  - 单元测试：TLS 状态 API 包含 acme_status 和 acme_error
  - 单元测试：ACME 重试逻辑

- **提交信息**
  `fix: improve ACME error handling and TLS status API feedback`
