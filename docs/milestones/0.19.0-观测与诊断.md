# 0.19.0: 观测与诊断

## Context

0.18.0 完成了配置备份与恢复功能（后端导出/导入/预览 API + 前端设置页备份面板）。REX Hub 目前缺少系统运行状态和连接性能的可观测性——用户无法直观看到资源连接延迟、吞吐量等性能趋势，也无法了解 Hub 自身的运行健康状况。本里程碑添加系统指标采集和可视化，帮助用户快速诊断连接异常和系统瓶颈。

版本类型：minor（新功能）

## 产品边界

**做什么：**
- 后端：各协议连接性能指标采集（延迟、吞吐量、错误率），存入数据库
- 后端：系统健康 API（CPU、内存、磁盘使用、数据库大小）
- 后端：指标聚合查询 API（按时间段、按资源聚合）
- 前端：仪表盘增加性能概览区域（连接延迟分布、资源健康度）
- 前端：资源详情页/工作区增加实时性能指标展示

**不做什么：**
- 不做第三方监控集成（Prometheus/Grafana 等）
- 不做长期指标存储和归档（只保留最近 7 天）
- 不做告警通知（邮件/Telegram/Webhook）
- 不做 Agent 侧指标上报（Agent 指标由 Hub 从心跳包中提取）
- 不做自定义仪表盘（使用固定布局）

## 子任务详细设计

### 1 指标数据模型与采集后端

**功能目标：**
在现有审计日志和心跳机制基础上，新增性能指标采集。通过心跳延迟、审计日志中的操作耗时、文件传输速率等已有数据源，提取性能指标并存入专用表。

**文件结构：**
```text
crates/rex-hub/src/
├── metrics.rs           新增：指标采集与存储逻辑
├── migrations.sql       修改：添加 metrics 表
├── db.rs                修改：添加指标相关方法
├── routes.rs            修改：注册指标路由
```

**数据模型：**
```sql
CREATE TABLE IF NOT EXISTS metrics (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    resource_id TEXT NOT NULL,
    metric_type TEXT NOT NULL,      -- 'latency' | 'throughput' | 'error' | 'connection'
    value REAL NOT NULL,            -- 指标值（延迟ms、吞吐量bytes/s、错误计数1）
    tags TEXT,                      -- JSON 扩展标签 '{"db":"mysql","query":"SELECT"}'
    recorded_at TEXT NOT NULL,      -- ISO 8601 时间戳
    FOREIGN KEY (resource_id) REFERENCES resources(id) ON DELETE CASCADE
);

CREATE INDEX idx_metrics_resource_type ON metrics(resource_id, metric_type);
CREATE INDEX idx_metrics_recorded_at ON metrics(recorded_at);
```

**采集来源：**
| 指标类型 | 来源 | 采集时机 |
|----------|------|----------|
| latency | SSH 心跳 RTT、SQL 查询耗时、SFTP 响应时间 | 每次操作时记录 |
| throughput | 文件传输速率、SQL 结果集大小 | 每次传输/查询完成时 |
| error | 连接失败、查询失败、认证失败 | 每次错误发生时 |
| connection | 连接建立/断开事件 | 每次连接状态变化时 |

**后端流程：**
1. 在 `ws_terminal`（SSH 心跳 RTT）、`sql`（查询耗时）、`files`（传输速率）、`ws_mysql/postgresql/redis/docker/sqlite/s3`（连接状态）中插入指标采集点
2. `metrics.rs` 提供 `record_metric()` 函数，批量写入指标
3. 后台定时任务：每小时清理超过 7 天的指标数据

**测试标准：**
- 指标写入数据库成功
- 按 resource_id 和 metric_type 查询正确
- 超过 7 天的数据自动清理


### 2 系统健康 API

**功能目标：**
实现系统健康检查 API，返回 Hub 运行状态、资源使用情况、数据库状态。

**文件结构：**
```text
crates/rex-hub/src/
├── health.rs            新增：系统健康检查逻辑
├── routes.rs            修改：注册健康路由
```

**API 设计：**
```text
GET /api/health
Response:
  {
    "status": "healthy",           // healthy | degraded | unhealthy
    "uptime_seconds": 86400,
    "version": "0.19.0",
    "database": {
      "size_bytes": 1048576,
      "tables": {
        "environments": 5,
        "resources": 12,
        "audit_log": 1500,
        "metrics": 28000
      }
    },
    "system": {
      "cpu_usage_percent": 12.5,
      "memory_usage_percent": 45.2,
      "disk_usage_percent": 67.8
    },
    "connections": {
      "agents_online": 3,
      "agents_total": 5,
      "active_sessions": 2
    }
  }
```

**后端流程：**
1. 读取进程运行时间
2. 查询数据库表行数和文件大小
3. 读取系统指标（`sysinfo` crate 或 `/proc` 文件系统）
4. 查询 Agent 在线数和活跃会话数
5. 根据各指标计算健康状态：
   - `healthy`：所有指标正常
   - `degraded`：CPU > 80% 或内存 > 85% 或磁盘 > 90%
   - `unhealthy`：数据库不可用或活跃会话为 0 但有 Agent 在线

**依赖：**
添加 `sysinfo` crate 到 workspace

**测试标准：**
- 返回完整的健康状态 JSON
- 数据库表行数统计正确
- 健康状态计算逻辑正确

**提交信息：**
```

```

### 3 指标聚合查询 API

**功能目标：**
实现指标聚合查询 API，支持按时间段、按资源、按指标类型查询聚合数据。

**文件结构：**
```text
crates/rex-hub/src/
├── metrics.rs           修改：添加聚合查询逻辑
├── routes.rs            修改：注册查询路由
```

**API 设计：**
```text
GET /api/metrics/summary
Query:
  resource_id: 可选，按资源筛选
  hours: 可选，默认 24（最近 N 小时）
Response:
  {
    "latency": {
      "avg_ms": 12.5,
      "p50_ms": 10.2,
      "p95_ms": 25.3,
      "p99_ms": 45.1,
      "min_ms": 2.1,
      "max_ms": 120.5
    },
    "throughput": {
      "avg_bytes_per_sec": 1048576,
      "total_bytes": 104857600
    },
    "errors": {
      "total": 5,
      "error_rate_percent": 0.5
    },
    "connections": {
      "total": 100,
      "successful": 95,
      "failed": 5
    },
    "recorded_at": "2026-06-25T10:00:00Z"
  }

GET /api/metrics/timeline
Query:
  resource_id: 可选
  metric_type: latency | throughput | error | connection
  hours: 可选，默认 24
  granularity: 可选，1h | 6h | 24h | 7d（默认 auto）
Response:
  {
    "data_points": [
      { "timestamp": "2026-06-25T08:00:00Z", "value": 12.5, "count": 50 },
      { "timestamp": "2026-06-25T08:15:00Z", "value": 15.2, "count": 48 }
    ],
    "metric_type": "latency",
    "unit": "ms"
  }
```

**后端流程：**
1. `GET /api/metrics/summary`：按时间范围查询指标，计算统计聚合（avg、p50、p95、p99、min、max）
2. `GET /api/metrics/timeline`：按时间窗口分桶，返回时间序列数据
3. 分桶策略：
   - 1h → 5 分钟粒度
   - 6h → 15 分钟粒度
   - 24h → 1 小时粒度
   - 7d → 6 小时粒度

**测试标准：**
- 聚合查询返回正确的统计值
- 时间线数据点数量与粒度匹配
- 空数据时返回零值而非错误

**提交信息：**
