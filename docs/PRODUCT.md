# REX Hub — 产品文档

REX Hub 是一款面向个人开发者和运维工程师的自托管远程资源统一管理平台。

---

## 1. 产品定位

### 核心问题

| 问题 | 现状 | REX Hub 的答案 |
|------|------|---------------|
| 多种资源需要多个工具 | SSH 用 Terminal/iTerm2，MySQL 用 Navicat/Workbench，SFTP 用 FileZilla… | **一个 Web 页面**管理所有远程资源 |
| 内网资源无法从外部访问 | 依赖 VPN、端口转发、跳板机 | 部署 Agent 后，**任何有网络的地方**都能访问 |
| 数据在第三方平台 | 使用在线 SaaS 工具，数据不可控 | 完全**自托管**，数据留在自己的基础设施上 |

### 两个目标

1. **一个页面访问多种资源** — SSH 终端、数据库查询、文件管理，不再在多个客户端之间切换。
2. **一个服务访问公网 + 内网资源** — 通过内网 Agent 代理，公网和内网服务器统一管理，无需 VPN 或端口转发。

### 产品属性

- **单用户** — 专为个人使用设计，不涉及多用户、团队协作、RBAC 权限。
- **自托管** — Docker 部署，数据完全自主掌控。
- **深色优先** — 终端美学，适合长时间运维工作场景。

## 2. 核心概念

### 2.1 环境 (Environment)

资源的逻辑分组。按部署网络或用途划分，每个环境拥有独立的 Agent 和资源集合。

典型分组：

| 环境 | 说明 |
|------|------|
| 阿里云 | 云服务器和数据库，直连访问 |
| 家里服务 | 家庭内网服务器和开发环境，通过 Agent 代理访问 |
| 家庭 NAS | 家庭内网存储设备，通过 Agent 代理访问 |

每个环境包含：
- 名称（必填）和描述（选填）
- 连接方式（Agent 代理 或 直连）
- 注册令牌（用于部署 Agent，绑定到该环境）
- 一个 Agent 实例（可选）
- 多个资源

### 2.2 Agent

部署在目标网络中的轻量级反向代理进程。

**工作方式：** Agent 从内网主动出站连接到 REX Hub 控制台，建立 WebSocket 加密隧道。内网服务器**无需开放任何入站端口**。

每个 Agent 属于一个环境，代理该环境下的所有内网资源。

**Agent 状态：**
- 🟢 在线 — Agent 已连接，资源可访问
- ⚫ 离线 — Agent 未运行或已断开，该环境的内网资源无法访问

**Agent 元信息：** 版本号、Agent ID、连接 IP、运行时间、延迟、传输量、代理资源数。

**Agent 设备信息：** Agent 上报运行环境的操作系统和架构，帮助用户了解远程环境特征，因为不同平台的终端操作存在差异：

| 字段 | 示例 | 说明 |
|------|------|------|
| 操作系统 | Linux / macOS / Windows | 运行 Agent 的主机系统 |
| 架构 | x86_64 / arm64 / armv7l | CPU 架构（影响二进制下载选择） |
| 主机名 | prod-server / raspberrypi | 远程主机名 |
| 系统版本 | Ubuntu 22.04 / macOS 14.3 / Windows Server 2022 | 详细 OS 版本 |

设备信息影响的功能：
- **终端提示**：SSH 连接时状态栏显示 `Linux · Ubuntu 22.04 · arm64`，帮助用户判断可用命令
- **平台差异提醒**：Windows 设备自动提示「Windows 环境，终端使用 PowerShell / cmd」
- **二进制下载**：部署指南根据 Agent 上报的 OS/架构自动推荐对应的下载命令
- **资源扫描**：Agent 根据 OS 类型自动发现可用服务（如 Linux 的 systemd 服务、Docker 容器）

### 2.3 资源 (Resource)

可连接的远程实体。每个资源属于一个环境，通过指定协议连接。

支持 8 种协议：

| 协议 | 图标 | 用途 | 色值 |
|------|------|------|------|
| SSH | `$` | 远程终端 | `#3FB950` 绿色 |
| SFTP | 📁 | 文件传输 | `#8B5CF6` 紫色 |
| MySQL | `dB` | 数据库查询 | `#58A6FF` 蓝色 |
| PostgreSQL | `pg` | 数据库查询 | `#8B5CF6` 紫色 |
| Redis | `R` | 缓存管理 | `#F85149` 红色 |
| Docker | 🐳 | 容器管理 | `#58A6FF` 蓝色 |
| SQLite | `S` | 本地数据库 | `#D29922` 橙色 |
| S3 / MinIO | ☁ | 对象存储 | `#E8912D` 主色 |

### 2.4 连接方式

| 方式 | 标识 | 说明 | 适用场景 |
|------|------|------|----------|
| Agent 代理 | ⬡ `#E8912D` | 通过内网 Agent 隧道访问 | 内网服务器、内网数据库 |
| 直连 | ◉ `#58A6FF` | 控制台直接连接目标 | 公网服务器、云数据库 |

### 3.4 资源创建

**页面：** `resource-new.html` · 标准布局

4 步向导，顶部进度指示器（圆点 + 连线）。

**第 1 步：选择协议**

协议网格，8 个选项卡片并排：
- 每个卡片：彩色图标 + 协议名 + 说明
- SSH（远程终端）、SFTP（文件传输）、MySQL（SQL 控制台）、PostgreSQL（SQL 控制台）、Redis（命令控制台）、Docker（容器管理）、SQLite（本地数据库）、S3 / MinIO（对象存储）
- 选中后高亮边框，底部「下一步」按钮激活

**第 2 步：基本信息 + 连接方式**

- 资源名称输入框，placeholder：「例如：云服务器、主数据库」
- 所属环境下拉框，选项为已有环境 + 「+ 创建新环境」
- 连接方式选择（两张卡片）：
  - ⬡ Agent 通过 Agent 代理 — 适用于内网资源，Agent 从内网主动连接服务器，无需开放入站端口
  - ◉ 直连服务器直连 — 适用于公网资源，REX Hub 服务器直接连接目标

**第 3 步：连接详情**

以 SSH 为例，不同协议显示不同表单：

SSH 连接表单：
- 主机地址（必填）、端口（默认 22）
- 用户名（必填）
- 认证方式切换：密码 / 密钥
  - 密码模式：密码输入框
  - 密钥模式：拖放/点击上传区域，支持 PEM、PPK、OpenSSH 格式
- 终端编码（UTF-8 推荐 / GBK / ISO-8859-1）
- 保活间隔（60s / 30s / 120s）

底部有「测试连接」按钮，点击后显示：
- 成功：绿色提示「连接成功！延迟 12ms，认证通过」
- 失败：红色提示「连接失败：Connection refused (port 22)」

**第 4 步：完成**

- 成功图标 ✓
- 「资源创建成功」标题
- 「资源已保存到 [环境名] 环境」说明
- 两个按钮：「立即连接」（跳转工作空间）、「返回仪表盘」

### 3.5 工作空间 (Workspace)

**页面：** `app.html` · 全屏工作区

多面板工作区，是核心操作界面。无侧边栏（通过全局快捷键和连接菜单操作）。

#### 标签页管理

- **标签栏** 位于顶部，横向排列所有已打开的标签
- 每个标签显示：协议图标（彩色）+ 资源名称 + 连接状态点（🟢 在线 / ⚫ 离线 / 🟠 连接中）+ 关闭按钮（悬停显示）
- 标签支持**拖拽排序**
- 右键/悬停标签关闭按钮可关闭标签
- **去重**：相同名称+协议的资源不会重复打开，跳转到已有标签
- 空状态时显示操作提示：`Ctrl+N` 新建连接、`Alt+1~5` 切换布局、`F1` 快捷键

#### 连接菜单

`Ctrl+N` 或点击 `+` 打开弹出式全局搜索菜单：

- 搜索框，支持名称、地址、协议模糊搜索
- 所有可用资源**按环境分组**展示
- 每个资源项：彩色协议图标 + 名称 + 地址 + 协议标签
- 键盘导航：↑↓ 选择、Enter 连接、Esc 关闭
- 底部提示操作方式

#### 分屏布局

5 种布局模式，通过 `Alt+1~5` 切换或点击右上角布局指示器：

| 布局 | 快捷键 | 面板数 | CSS 标识 |
|------|--------|--------|----------|
| 单面板 | `Alt+1` | 1 | `layout-single` |
| 左右分屏 | `Alt+2` | 2 | `layout-left-right` |
| 上下分屏 | `Alt+3` | 2 | `layout-top-bottom` |
| 四宫格 | `Alt+4` | 4 | `layout-quad` |
| 主 + 侧边 | `Alt+5` | 2 | `layout-sidebar-main` (2:1) |

- 双击标签可快速进入左右分屏
- 布局切换时，按标签顺序自动填充面板

#### 面板拖拽

支持将标签拖拽到不同的分屏面板位置：

- **拖拽标签到面板**：拖住标签拖向目标面板区域，目标面板显示虚线边框高亮（`layout-drop-zone`），松开后标签内容填充到该面板
- **面板间交换**：将标签拖到已占用的面板上，交换两个面板的内容
- **拖出分离**：将标签拖出面板区域，可在新窗口中打开（可选功能）
- **视觉反馈**：拖拽过程中，源标签降低透明度（`opacity: 0.5`），目标面板显示橙色虚线边框 + 半透明背景提示
- **规则**：单面板模式下不支持面板拖拽；分屏模式下，拖入空面板自动填充，拖入已占用面板则交换内容

#### 快捷键面板

`F1` 打开快捷键帮助面板（模态弹窗），分组展示所有快捷键。

**全屏/窗口控制：** `F11` 全屏切换

#### 标签右键菜单

标签栏中右键点击某个标签弹出上下文菜单：

| 菜单项 | 说明 |
|--------|------|
| 关闭 | 关闭当前标签 |
| 关闭其他 | 关闭除当前标签外的所有标签 |
| 关闭右侧 | 关闭当前标签右侧的所有标签 |
| 关闭左侧 | 关闭当前标签左侧的所有标签 |
| 关闭全部 | 关闭所有已打开的标签 |
| ─── 分隔线 ─── | |
| 复制标签 | 复制当前连接，打开同资源的新标签 |
| 移动到面板 ▸ | 子菜单：面板 1 / 面板 2 / 面板 3 / 面板 4（仅分屏模式可用） |
| ─── 分隔线 ─── | |
| 新建连接 | 打开连接菜单 |
| 全部断开 | 断开所有标签的连接 |

### 3.6 SSH 终端

**页面：** `terminal.html` · 全屏

基于 xterm.js 的完整终端界面。

**标签栏：** 多标签管理，支持新建、关闭、切换。

**工具栏：**
- 左侧：连接状态指示（🟢）+ 资源名称 + 延迟（ms）
- 右侧操作按钮：📋 复制、📑 粘贴、🗑 清屏、📁 SFTP、⛶ 全屏、断开连接

**内置 SFTP：**
SSH 终端内置 SFTP 文件管理能力，无需单独创建 SFTP 资源：
- 工具栏「📁 SFTP」按钮：点击后在右侧/下方打开内嵌 SFTP 面板（split view），共享同一个 SSH 连接
- SFTP 面板可拖拽调整大小、独立浏览远程文件系统
- 从 SFTP 面板拖拽文件到终端区域，自动将文件路径粘贴到终端
- SSH 工具栏显示「📁 SFTP」按钮时，表示当前 SSH 连接支持文件传输
- SFTP 资源仍可作为独立协议单独创建和使用（适用于只需文件管理的场景）

**终端区：**
- monospace 字体（JetBrains Mono），13px
- 完整 ANSI/VT100 仿真
- 256 色和 TrueColor 支持
- 闪烁光标动画

**状态栏（底部橙色条）：**
- 左侧：协议（SSH）· 字符编码（UTF-8）· 终端尺寸（180×42）
- 右侧：连接方式（通过 Agent 代理）· 操作提示（Ctrl+Shift+C 复制）

**终端区右键菜单：**

| 菜单项 | 说明 |

| ─── 分隔线 ─── | |
| 打开 SFTP 面板 | 在右侧/下方打开内嵌 SFTP 文件管理面板 |
| 在新标签中打开 SFTP | 为当前 SSH 连接打开独立 SFTP 标签页 |
| ─── 分隔线 ─── | |
| 新建 SSH 连接 | 打开连接菜单，快速新建标签 |

#### 部署指南

1. 创建 `docker-compose.yml` — 完整编排文件示例（image、restart、environment、volumes、network_mode、logging）
2. 创建 `agent.yaml`（可选）— 高级配置
3. 启动/停止/更新命令
4. 确认连接

**方式 4：配置文件**
1. 创建 `agent.yaml` — server、token、name 字段
2. 使用配置启动：`./rex-agent --config agent.yaml`

所有代码块带「复制」按钮。

## 8. 技术架构

### 进程模型

Hub 和 Agent 使用相同的运行时模型：单个可执行文件启动后，父进程作为 supervisor，启动 1 个 worker 子进程负责业务逻辑。

```text
rex-hub / rex-agent 启动（PID 1）
  ↓
父进程进入 supervisor 模式
  ↓
启动 worker 子进程
  ├── worker：执行业务逻辑、维护连接、检查更新、写入状态
  └── supervisor：启动 worker、监控存活、根据状态执行替换或回滚
```

父进程就是 supervisor，不需要 s6-overlay、launcher 或 wrapper。第一阶段不自动更新时，父进程只做启动和监控；第二阶段增加更新检测、替换和回滚能力，进程模型不变。

更底层的入口逻辑、状态文件、退出码、Docker 信号处理和 Windows 差异见 [DEVELOPMENT.md](DEVELOPMENT.md)。

### 自更新机制

#### 阶段 1：不自动更新

Hub 和 Agent 均为单二进制文件，不自动更新，由用户手动替换二进制或重新拉取 Docker 镜像完成升级。

第一阶段仍采用 supervisor + worker 模型，但父进程只负责启动和监控，不涉及更新流程、回滚机制或签名校验。

#### 阶段 2：自动更新

第二阶段在同一个进程模型上增加自动更新能力。

**Hub 更新流程：**

```text
worker 检查 GitHub Releases
  ↓
下载新二进制 → 保存为 staging
  ↓
校验 SHA256
  ↓
签名无效 → 删除 staging，继续运行旧版
  ↓
签名有效 → 备份当前二进制为 rollback
  ↓
写入 update-state.json：phase=requested
  ↓
worker 正常退出，请求 supervisor 重启
  ↓
supervisor 替换二进制：staging → 当前二进制
  ↓
启动新 worker，并传入 REX_UPDATE_PENDING=1
  ↓
新 worker 轮询 /healthz，30 秒内健康？
  ├─ 是 → 更新完成，删除 rollback
  └─ 否 → 失败次数 + 1，连续 3 次后恢复 rollback
```

**Agent 更新流程：**

```text
worker 连接 Hub 时上报版本号 + SHA256
  ↓
Hub 校验签名不匹配 → 返回「需要更新」+ 下载 URL
  ↓
worker 从 Hub 下载新二进制：GET /api/agent/download?os={os}&arch={arch}
  ↓
校验 SHA256
  ↓
签名无效 → 删除 staging，继续运行旧版
  ↓
签名有效 → 备份当前二进制为 rollback
  ↓
写入 update-state.json：phase=requested
  ↓
worker 正常退出，请求 supervisor 重启
  ↓
supervisor 替换二进制：staging → 当前二进制
  ↓
启动新 worker，并传入 REX_UPDATE_PENDING=1
  ↓
新 worker 30 秒内连接 Hub 成功？
  ├─ 是 → 更新完成，删除 rollback
  └─ 否 → 失败次数 + 1，连续 3 次后恢复 rollback
```

**关键设计：**
- 父进程保持稳定，只负责启动 worker、监控存活、读取状态、替换二进制和回滚。
- worker 负责业务逻辑、更新检测、下载、校验、备份和状态写入。
- 新 worker 启动时由父进程传入 `REX_UPDATE_PENDING=1` 和 `REX_TARGET_VERSION`，避免重复检查更新导致死循环。
- 更新只替换二进制，Agent 的 `agent.json` 保存在持久化数据目录中，升级后仍保持同一 Agent 身份。
- Hub 和 Agent 版本必须一致，不存在跨版本兼容；每个 Hub 版本的部署包内打包对应版本的 Agent 二进制。

Docker 内同样使用这一套 supervisor + worker 模型，PID 1 就是二进制本身，不需要 s6-overlay。Windows 下需要运行时 supervisor 副本以避免 `.exe` 运行中被锁定，用户入口仍然是单个命令。更多实现细节见 [DEVELOPMENT.md](DEVELOPMENT.md)。

### 共享 Rust crate

代码从第一阶段开始按 crate 组织，Hub 和 Agent 共享协议与传输能力。第一阶段编译成稳定的单二进制；第二阶段增加 supervisor 更新能力时，无需改变产品形态。

```text
rex-ssh          SSH/SFTP 协议实现
rex-mysql        MySQL 协议实现
rex-postgresql   PostgreSQL 协议实现
rex-redis        Redis 协议实现
rex-docker       Docker 协议实现
rex-sqlite       SQLite 协议实现
rex-s3           S3/MinIO 协议实现
rex-transfer     文件传输引擎
rex-tunnel       WebSocket 隧道
rex-supervisor   进程 supervisor（启动、监控、状态判断、替换、回滚）
```

Hub 和 Agent 使用同一套 crate：

```text
rex-hub   = 协议 crate + 文件传输 + WebSocket 隧道 + supervisor + Web 前端
rex-agent = 协议 crate + 文件传输 + WebSocket 隧道 + supervisor
```

### 文件传输架构

**前端：** 独立 Vue 包，由 Hub 静态托管
- 文件浏览器 UI、传输队列 UI、进度展示、冲突处理弹窗
- 只调用 Hub API / WebSocket，不参与数据中转

**后端：** `rex-transfer` crate
- 负责任务调度、进度、取消、重试和校验
- 通过统一文件接口对接 SSH/SFTP、S3/MinIO、Docker、本地文件等连接器
- 数据在 Hub / Agent / 远端资源之间传输，不经过浏览器中转

**冲突处理：**
- 目标不存在：直接写入
- 目标已存在：覆盖、跳过、另存为、失败

**安全限制：**
- 路径规范化，禁止 `../` 逃逸
- 禁止访问未授权路径
- 限制最大文件大小、并发数、传输速率
- 所有文件操作写入审计日志

## 9. 部署

### 控制台（Docker）

```bash
docker run -d \
  --name rex-hub \
  -p 3000:3000 \
  -v ./data:/app/data \
  -e REX_SECRET_KEY=your-secret-key \
  rexhub/console:latest
```

访问 `http://localhost:3000` 开始使用。

容器内 `rex-hub` 作为 PID 1，自动启动 worker 子进程监听端口。

### Agent（Docker）

```bash
docker run -d \
  --name rex-agent \
  --restart unless-stopped \
  -e REX_SERVER=https://your-hub.com \
  -e REX_TOKEN=<环境注册令牌> \
  -v ./agent-data:/data \
  -v /var/run/docker.sock:/var/run/docker.sock \
  rexhub/agent:latest
```

> `./agent-data:/data` 挂载数据卷，持久化 Agent 身份（`agent.json`），容器重建或更新后保持同一 Agent ID。

容器内 `rex-agent` 作为 PID 1，自动启动 worker 子进程连接 Hub。

### Agent（二进制）

```bash
# 下载
curl -fsSL https://get.rexhub.dev/agent/linux-amd64 -o rex-agent
chmod +x rex-agent

# 启动
# 或手动指定数据目录
./rex-agent \
  --server https://your-hub.com \
  --token <环境注册令牌> \
  --name "内网 Agent" \
  --data-dir ./agent-data
```

> `--data-dir ./agent-data` 指定数据目录，含 `agent.json`（身份持久化）和更新状态/回滚文件。

```yaml
# agent.yaml（可选的高级配置）
server: https://your-hub.com
token: <环境注册令牌>
name: "内网 Agent"
data-dir: "./agent-data"
auto-update: true
```

### Agent（配置文件）

```yaml
# agent.yaml
server: https://your-hub.com
token: <环境注册令牌>
name: "内网 Agent"
data-dir: "./agent-data"   # 可选，默认为 ./data
auto-update: true          # 可选，默认为 true
```

```bash
# 启动
./rex-agent --config agent.yaml
```

注册令牌在对应环境的详情页面获取。
